SSH와 SSL/TLS는 둘 다 공개키 암호를 기반으로 안전한 채널을 만들지만, SSH는 "원격 셸 접속을 위한 완결된 응용 프로토콜"이고 TLS는 "어떤 응용 프로토콜이든 감싸는 범용 암호화 래퍼"다. 같은 암호학 재료를 쓰되 조립 방식이 완전히 다르며, SSH가 SSL 위에서 도는 관계도 아니다.

## 목적이 다르다

SSH(Secure Shell)는 원격 서버에 접속해 명령을 실행하기 위한 프로토콜이다. 서버 관리, 원격 로그인, 파일 전송(SCP/SFTP), 포트 포워딩이 주 용도이고 기본 포트는 22번이다. "사람이 서버 안으로 들어가 직접 작업하는 보안 통로"에 해당한다.

SSL/TLS는 클라이언트-서버 간 데이터 전송 자체를 암호화하는 프로토콜이다. HTTPS(웹), 메일(SMTPS/IMAPS), DB 연결 등 다양한 응용 통신을 보호한다. "데이터가 오갈 때 그 내용을 감싸는 보안 봉투"다. SSL은 취약점으로 폐기되고 지금은 TLS가 표준이다.

## 계층 구조가 다르다

둘 다 OSI 기준으로 응용 계층(L7) 영역에서 TCP 위에 얹히지만, 위치와 구조가 다르다.

TLS는 TCP 바로 위, 응용 프로토콜 바로 아래에 놓이는 중간 보안 계층이다(교과서에서는 L5~L6 사이로 설명하기도 한다). HTTP·SMTP·FTP 같은 상위 프로토콜이 그대로 TLS 위에 얹혀 동작한다(HTTP → HTTPS). 응용 프로토콜과 독립적이라 어떤 애플리케이션이든 그 아래에 끼워 넣을 수 있다.

```
[Application: HTTP, SMTP ...]
[TLS]
[TCP]
[IP]
```

SSH는 완전한 응용 계층 프로토콜이지만 내부에 자체 계층 구조를 품는다.

```
[SSH Connection Layer]     ← 채널 다중화 (셸, 포트포워딩)
[SSH Authentication Layer] ← 사용자 인증
[SSH Transport Layer]      ← 암호화·무결성·서버 인증 (SSH 자체 정의)
[TCP]
[IP]
```

여기서 "SSH Transport Layer"는 OSI 전송 계층(L4)이 아니라 SSH가 자체 정의한 내부 계층이다 — 이름 때문에 헷갈리기 쉬운 지점이다.

핵심은 이 구조 차이다. TLS는 응용 프로토콜과 분리된 범용 보안 래퍼라 여러 프로토콜이 공유해서 쓴다. SSH는 암호화·인증·채널 관리까지 하나의 통합된 응용 프로토콜 안에 묶여 그 자체로 완결된 원격 접속 시스템이다.

## 인증 방식이 다르다

SSH는 주로 사용자 중심 인증이다. 사용자가 자체 생성한 키 쌍(또는 비밀번호)으로 자신을 증명한다. CA가 개입하지 않는다.

TLS는 주로 서버 중심 인증이다. 서버가 CA(인증기관)가 발급한 인증서를 제시해 신원을 증명한다. 브라우저가 HTTPS 사이트의 정체를 신뢰하는 근거가 이 CA 체인이다.

## 흔한 혼동: SSH는 SSL 기반인가?

아니다. SSH는 SSL/TLS를 전혀 쓰지 않는다. TCP 위에서 곧바로 자기만의 핸드셰이크와 키 교환·암호화(SSH Transport Layer Protocol)를 수행하며, 중간에 TLS 계층이 끼어들지 않는다.

```
SSH:    [SSH 자체 암호화] → [TCP] → [IP]
HTTPS:  [HTTP] → [TLS] → [TCP] → [IP]
```

둘이 비슷해 보이는 이유는 같은 암호학적 재료를 공유하기 때문이다. 키 교환에 Diffie-Hellman(ECDH), 대칭 암호화에 AES, 무결성에 HMAC, 인증에 RSA·ECDSA·Ed25519 같은 공개키 알고리즘을 양쪽 다 쓴다. 재료는 같지만 조립 방식(프로토콜)은 완전히 다르다.

역사적으로도 SSH(1995)와 SSL(1995)은 비슷한 시기에 서로 다른 주체가 독립적으로 만들었고, 어느 한쪽이 다른 쪽 위에 세워진 관계가 아니다.
